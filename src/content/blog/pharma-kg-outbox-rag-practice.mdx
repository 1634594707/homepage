# 医药知识图谱平台实战：Outbox 双库一致 + RAG 主链路

> **TL;DR**  
> 医疗场景下，系统价值来自“一致性 + 可追溯 + 风控闭环”，而不只是回答速度。

这篇文章记录我在医药智能知识图谱平台中的三块核心工作：

1. PostgreSQL 到 Neo4j 的双库一致；
2. RAG 问答主链路；
3. 高风险场景下的审核与风控。

---

## 一、为什么要做双库一致

业务数据在关系库，图谱查询在 Neo4j。  
如果写入不同步，就会出现“后台有数据，图里查不到”或者“图里有旧数据”的问题。

我采用的是 Outbox 思路：

- 业务事务提交时，同时写一条 Outbox 事件；
- 事务提交后由异步消费者处理；
- 消费端写 Neo4j 时使用 `MERGE` 保证幂等；
- 失败重试并记录审计日志。

这样做的收益是：最终一致可保证、失败可追踪、恢复路径清晰。

## 二、RAG 主链路怎么搭

我的主流程是：

```text
实体抽取 -> 参数化 Cypher 检索 -> 结构化证据注入 -> 受控生成 -> 后处理
```

这里有两个关键细节：

- 检索结果要“结构化返回”，而不是直接拼文本；
- 生成阶段要限制边界，证据不足时必须显式说明。

## 三、检索性能优化思路

做法比较朴素，但很有效：

- 沉淀高频 Cypher 模板，减少临时拼接和不稳定查询；
- 对关键字段建立索引；
- 对查询结果做 TopN 与返回规模限制；
- 热点问题引入 Redis Cache-Aside。

此外我加了 `graph_cache_version` 版本失效策略，避免图谱更新后缓存“假新鲜”。

在核心接口上，这套优化让平均响应时间下降约 60%。

## 四、医疗场景下的风险控制

医药问答不能只追求“答得快”，更要“答得稳”。  
我在链路中加入了风控闭环：

- 先做风险打分（`riskScore`）；
- 超阈值进入医生审核队列；
- 输出前强制免责声明与安全清洗。

这让系统在复杂问题上更可控，也更符合真实业务要求。

## 五、我的复盘

这个项目让我确认了一件事：  
后端 + AI 的价值不在“调用模型”，而在把模型能力纳入工程系统，做到可控、可追溯、可恢复。

如果只看单次回答，很多方案都能跑；  
但要长期稳定服务真实用户，架构细节才是胜负手。

---

## 关键结论

- 用 Outbox 保障双库最终一致，先解决“数据可信”。
- 用结构化证据注入，提高问答可追溯性。
- 用风险评分与审核队列，守住医疗场景安全边界。
