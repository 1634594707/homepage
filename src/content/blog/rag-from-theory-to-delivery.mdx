# RAG 从原理到落地：我的工程化实践

> **TL;DR**  
> RAG 的关键不在“接上模型”，而在检索稳定、证据可追溯、失败可控制。

RAG（Retrieval-Augmented Generation）不是“加个向量库”就结束，而是一条完整的工程链路：文档处理、向量化、检索策略、上下文组装、模型生成、风险控制和效果评估。

---

## 一、标准流程

```text
文档上传 -> 解析与分块 -> Embedding -> 向量存储
用户提问 -> Query Embedding -> TopK 检索 -> Prompt 组装 -> LLM 生成
```

在项目里，我更关注两件事：

1. 回答能不能追溯到证据；
2. 高并发时链路是否稳定。

## 二、分块策略比模型更容易被忽视

常见分块方式有三种：

- 固定长度分块：简单，但容易切断语义；
- 按标题分块：适合 Markdown 和结构化文档；
- 语义分块：效果更好，但成本更高。

在 `AgentFlow` 中我优先使用“按章节分块”，因为它在“可解释性”和“实现复杂度”之间平衡得更好。

## 三、向量存储：为什么我常用 pgvector

很多同学会问：为什么不用专门向量数据库？

我的实际选择是看系统目标：

- 如果系统已经以 PostgreSQL 为主，`pgvector` 能减少组件复杂度；
- 结构化数据和向量数据同库，事务一致性更容易保障；
- 对中小规模知识库（如 10 万级别）性能通常够用。

示例查询（核心思想）：

```sql
SELECT content, embedding <-> :query_vector AS distance
FROM chunk_bge_m3
WHERE kb_id = :kb_id
ORDER BY embedding <-> :query_vector
LIMIT 5;
```

## 四、检索优化的优先级

我常用的优化顺序：

1. 先把 `TopK`、过滤条件（如 `kb_id`）做正确；
2. 再做重排序（rerank）提升精度；
3. 最后做混合检索（向量 + 关键词）扩召回。

这比一开始就上复杂方案更稳。

## 五、Prompt 约束与安全边界

对于医疗等高风险场景，我会把规则写进系统提示词：

- 仅依据检索证据回答；
- 证据不足要明确说明；
- 输出中加入风险提示和免责声明；
- 禁止超出知识边界的结论性建议。

这类约束不能完全消除风险，但能显著降低“看起来很对但其实不对”的回答比例。

## 六、落地复盘

RAG 不是单点能力，而是工程系统能力。  
真正决定体验的，往往不是“模型再强一点”，而是：

- 检索是否稳定；
- 证据是否可追溯；
- 失败路径是否可控。

对后端工程师来说，把这些做扎实，比追逐新名词更有长期价值。

---

## 关键结论

- 先把检索链路做稳，再追求复杂策略。
- 先保证证据可追溯，再优化生成效果。
- 先明确系统边界，再谈模型能力上限。
