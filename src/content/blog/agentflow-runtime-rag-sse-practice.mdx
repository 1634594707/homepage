# AgentFlow 实战复盘：Runtime、RAG 与 SSE 稳定性

> **TL;DR**  
> AgentFlow 的核心不是“多会话对话”，而是可控 Runtime、稳定检索与高并发推送治理。

`AgentFlow` 是我做过的一个 AI Agent 后端项目。  
系统目标不只是“能对话”，而是把 Agent 能力做成可控、可观测、可演进的工程系统。

---

## 一、核心链路

整体数据流可以概括为：

```text
前端提问 -> 会话持久化 -> Agent Loop -> 工具/知识库 -> SSE 实时推送
```

其中最关键的是 Agent Runtime 的循环控制，而不是单次模型调用。

## 二、Think-Execute-Loop 的落地思路

我把运行时拆成三个阶段：

1. **Think**：模型基于上下文决定下一步（是否调用工具）；
2. **Execute**：服务端统一执行工具调用并回灌结果；
3. **Loop**：继续迭代，直到任务完成。

为了防止死循环，我做了两层限制：

- `MAX_STEPS` 最大步数上限；
- `terminate` 终止工具显式收敛。

## 三、为什么关闭自动工具执行

项目里我关闭了框架默认自动执行工具的行为，改为服务端统一调度。这样能把治理能力插在统一入口：

- 审计与日志；
- 鉴权与白名单；
- 超时、重试、限流；
- 失败降级与异常隔离。

这让工具调用从“黑盒动作”变成“可治理能力”。

## 四、RAG 检索链路与 pgvector

在知识库检索上，我采用 `PostgreSQL + pgvector`：

- chunk 向量化后存储 `vector(1024)`；
- 查询时按 `kb_id` 过滤，再做 TopK 相似检索；
- 把证据片段注入上下文，提升回答可追溯性。

示例检索思想：

```sql
SELECT content
FROM chunk_bge_m3
WHERE kb_id = :kb_id
ORDER BY embedding <-> :query_vector
LIMIT :k;
```

## 五、向量检索优化：从“能查”到“稳查”

仅靠排序检索在数据量上升后容易出现 P95/P99 抖动。  
我在优化里重点关注：

- ANN 索引（HNSW / IVFFLAT）；
- 参数调优（如 `ef_search`、`lists`、`probes`）；
- 执行计划对比与慢查询分析。

目标不是只看均值，而是让尾延迟更稳定。

## 六、SSE 高并发稳定性

对实时推送链路，我重点做了三件事：

- 连接生命周期清理（completion / timeout / error）；
- 异步线程池隔离，避免阻塞请求线程；
- 推送失败与主流程隔离，避免级联失败。

这三点看起来基础，但在高并发场景下决定了系统是否“长期可跑”。

## 七、复盘

这个项目让我形成了一个明确认知：  
Agent 后端的核心竞争力是“工程治理能力”，包括可控收敛、权限边界、检索稳定性与链路可观测性。

模型能力很重要，但把模型纳入工程系统，才是可交付的产品能力。

---

## 关键结论

- 关闭自动工具执行，统一服务端调度更可控。
- RAG 关注 P95/P99，比只看平均值更真实。
- SSE 稳定性取决于连接清理、线程隔离和失败隔离。
