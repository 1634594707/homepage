# Agent Loop 设计笔记：让智能体可控地跑起来

> **TL;DR**  
> Agent 的核心不是“会调用工具”，而是“可控地调用工具”。

做 Agent 系统时，我最早踩的坑是“能跑但不可控”。  
后来把链路改成 `Think -> Execute -> Loop`，并增加状态机与终止机制，系统才真正可用于线上场景。

---

## 一、核心循环

```text
Think: 模型判断下一步动作
Execute: 服务端执行工具调用
Loop: 将结果回灌上下文并进入下一轮
```

关键点是：工具执行权必须在服务端，而不是完全交给模型自动执行。

## 二、为什么禁用自动工具执行

我在实践中会关闭框架默认的自动执行，原因有三：

- 便于插入审计、限流、超时、重试；
- 便于做工具白名单（`allowed_tools`）；
- 便于在执行失败时把错误上下文回灌模型，让它修正策略。

这让 Agent 从“黑盒行为”变为“可观测系统”。

## 三、状态机是稳定性的基础

一个最小可用状态集：

- `IDLE`
- `THINKING`
- `EXECUTING`
- `FINISHED`
- `ERROR`

状态明确后，SSE 推送、前端展示、异常恢复都会简单很多。

## 四、必须有的三道防线

### 1) 最大步数限制

设置 `MAX_STEPS`，防止死循环。

### 2) 显式终止工具

提供 `terminate` 工具，让模型能主动结束任务并返回结束原因。

### 3) 全链路超时

给单次任务设置整体超时，避免长尾请求占满资源。

## 五、工具体系如何抽象

我常用“固定工具 + 可选工具”的结构：

- 固定工具：知识检索、直接回答、终止；
- 可选工具：数据库、外部 API、文件处理等。

再配合白名单做按 Agent 的能力裁剪。  
这样既能扩展，也能把风险边界控制住。

## 六、落地建议

如果你也在做 Agent 后端，我建议优先保证四件事：

1. 可观测（状态、日志、事件）；
2. 可控（步数、超时、白名单）；
3. 可回放（保存执行轨迹）；
4. 可恢复（失败后可重试或安全退出）。

先把“系统性”做出来，再追求“更聪明”的效果，会更稳。

---

## 关键结论

- 把工具执行放在服务端，才能做治理。
- 没有状态机与终止策略，Agent 很难上线。
- 先追求稳定收敛，再追求复杂能力扩展。
